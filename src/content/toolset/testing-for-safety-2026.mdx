A sanitizer instrumenting code that nobody calls is silent. The whole "ASan + UBSan in CI" story (<PostLink slug="sanitizers-2026">sanitizers-2026</PostLink>) only catches bugs in code paths the test suite actually exercises. **Tests are the input; sanitizers are the lens.** This page covers the input side: the four coverage levels that make the lens worth running, the reflection-driven shortcut that keeps the test suite in sync with the schema as the codebase grows, and the C++26 contracts + C++29 injection direction that makes the spec the test.

## Today

### Four coverage levels, ordered by what they catch

| Level | Tool | What it catches | Cost |
|---|---|---|---|
| **Example-based** | [Catch2 v3](https://github.com/catchorg/Catch2), [doctest](https://github.com/doctest/doctest), [GoogleTest](https://github.com/google/googletest) | Regressions in known scenarios; named happy-path + failure-path per public API | Low: write per case |
| **Property-based** | [RapidCheck](https://github.com/emil-e/rapidcheck) (on Catch2 / GTest) | Corners example-based misses; algorithmic invariants ("for all inputs, parse(serialize(x)) == x") | Medium: write the invariant once, library generates inputs |
| **Coverage-guided fuzzing** | [libFuzzer](https://llvm.org/docs/LibFuzzer.html) (bundled in `cpp-safety`), [AFL++](https://aflplus.plus/) | UB the example + property tests don't reach; trust-boundary parser bugs | Medium per harness; ~30min per harness usually finds bugs that survived years of human-written tests |
| **Differential** | Two implementations + a runner | Cross-impl divergence; spec disagreements; refactor regressions | High setup; lowest false-positive rate |

The minimum-viable safety story: **example-based + sanitizers in every PR**. Add property-based and fuzzing as the codebase matures. Differential testing is reserved for "two impls of the same thing exist" -- a state-machine refactor against the previous release, a fast-path against the slow correct path, a custom JSON parser against `nlohmann::json`.

### Coverage measurement

Sanitizer-clean CI without coverage data proves nothing. Wire `gcov` / `llvm-cov` (clang) or `lcov` into your CI as the third leg of the stool:

```cmake
# CMake snippet for coverage instrumentation
set(WROCPP_COV "off" CACHE STRING "Coverage backend: gcov|llvm|off")
set_property(CACHE WROCPP_COV PROPERTY STRINGS off gcov llvm)

if(WROCPP_COV STREQUAL "gcov")
    add_compile_options(--coverage -O0 -g)
    add_link_options(--coverage)
elseif(WROCPP_COV STREQUAL "llvm")
    add_compile_options(-fprofile-instr-generate -fcoverage-mapping -O0 -g)
    add_link_options(-fprofile-instr-generate)
endif()
```

CI step: build with coverage, run the test suite, generate the report (`lcov --capture --output-file coverage.info` for gcov, `llvm-profdata merge` + `llvm-cov` for clang). Fail the build if line coverage drops below the team's floor; 80% is a reasonable starter (regulated industries push this to 95%+ with MC/DC for ASIL-D).

### When to reach for which framework

| Framework | Best when | Cost |
|---|---|---|
| **Catch2 v3** | Header + linked, BDD-friendly assertions, broad ecosystem. Default for greenfield. | Long compile (link required) |
| **doctest** | Single-header, fast compile, drop-in for header-only libs. | Smaller community |
| **GoogleTest** | Established; integrates with GoogleMock for ergonomic mocks; corporate-friendly. | Bazel-aligned; CMake support is good but verbose |
| **RapidCheck** | Property-based on top of any of the above. | Auxiliary; not a standalone framework |
| **libFuzzer** | In-process, coverage-guided. The default fuzz target for new harnesses. | Per-harness setup |
| **AFL++** | Out-of-process, slow but covers more file-format complexity. | Heavier infra |

For automotive / aerospace / medical, the framework choice is often dictated by the certified toolchain. The pattern below (reflection-driven test generation) is framework-agnostic -- the per-member loop becomes a Catch2 `SECTION`, a GoogleTest `TEST_P`, or a doctest `SUBCASE`.

## Reflection today (C++26, clang-p2996 + GCC 16.1)

A test that writes `42` to a field and reads it back tests C++ assignment, not the production code. The bugs that hit production are different shapes: the parser and serializer drift apart after one of them gains a field; an interface gains a method but only half its mocks know; a "this should never happen in the happy path" log line starts firing because nobody asserted the call count. Reflection (P2996) and **user-defined annotations (P3394, also adopted into C++26)** together let the schema itself carry the test specification: per-field hints constrain the generators; struct-level markers dispatch which properties to assert. The runner is generic; the spec lives next to the data.

### Annotation-driven round-trip property test

The parser/serializer pair from <PostLink slug="sanitizers-2026">sanitizers-2026</PostLink> has three real invariants -- the kind that break silently in production when one side of the pair is edited and the other is not:

1. `parse(serialize(x)) == x` for every well-formed value `x`.
2. `serialize(parse(b)) == b` for every byte stream of length `wire_size<T>()`.
3. `parse(b)` refuses cleanly when `b.size() < wire_size<T>()`.

The annotation taxonomy borrows from established property-based testing libraries -- nothing here is novel, just the C++ syntax. The cousins, side by side:

| Our annotation | hypothesis (Python) | JUnit 5 (Java) | proptest (Rust) | kotest / jqwik (Kotlin) |
|---|---|---|---|---|
| `[[=test_range{lo,hi}]]` on field | `st.integers(min_value=lo, max_value=hi)` | (no native; emulate via `@CsvSource`) | `#[strategy(lo..hi)]` | `Arb.int(lo..hi)` |
| `[[=test_examples{{...}, n}]]` on field | `@example(value)` | `@ValueSource(ints={...})` | `prop_oneof![Just(a), Just(b)]` | `Exhaustive.of(...)` |
| `[[=test_round_trip{}]]` struct marker | `@settings(...)` + property body | `@Tag("round_trip")` on class | `#[proptest]` fn attribute | `@Property` (jqwik) |
| `[[=test_reject_short{}]]` struct marker | (separate test fn) | `@Tag` | (separate fn) | (separate spec) |

Field-level annotations describe the valid-input domain the generator should exercise; struct-level markers dispatch which properties to assert. Same shape as `@derive(Arbitrary)` in proptest -- the type IS the test specification.

```cpp
struct test_range    { long long lo, hi; };
struct test_examples { long long v[4]; std::size_t n; };  // pinned cases
struct test_round_trip {};
struct test_reject_short {};

struct [[=test_round_trip{}]] [[=test_reject_short{}]] SensorReading {
    [[=test_range{0, 1023}]]
    [[=test_examples{{0, 1023, 1024, 0}, 3}]]   // lo, hi, hi+1 (off-by-one)
    std::uint16_t sensor_id;

    [[=test_range{-100000, 100000}]]
    std::int32_t  raw_value;

    [[=test_range{0, 0xFF}]]
    std::uint8_t  status_flags;
};
```

The runner reads annotations back via `std::meta::annotations_of(m)` + `std::meta::extract<Tag>(a)`. Per field, it builds a sample set from `test_range` (`{lo, mid, hi}`) plus any `test_examples` (pinned values, hypothesis-style) and runs the cross-product through the parse/serialize pair:

```cpp
template <typename T>
auto run_annotated_tests() -> bool {
    auto samples = generate_samples<T>();   // cross-product per [[=test_range]]
    bool ok = true;
    if constexpr (has_annotation<test_round_trip>(^^T)) {
        ok &= check_value_round_trip<T>(samples);   // P1
        ok &= check_byte_round_trip<T>(samples);    // P2
    }
    if constexpr (has_annotation<test_reject_short>(^^T)) {
        ok &= check_truncated_refused<T>();         // P3
    }
    return ok;
}
```

For `SensorReading` (sensor_id has 3 range + 3 pinned = 6 samples, raw_value has 3, status_flags has 3 -- cross-product is 54), the runner emits:

```
== annotation-driven tests for SensorReading ==
== 54 samples generated from per-field annotations ==
  P1 value round-trip (54 samples, 0 failed): PASS
  P2 byte round-trip (54 samples): PASS
  P3 truncated refused (lengths 0..6): PASS
--
overall: PASS
```

Add a fourth field with its own `[[=test_range]]`, the cross-product expands automatically and the next CI run exercises 162 samples. Swap two field declarations by mistake (a real bug: parser reads `u16` then `i32` but serializer writes `i32` then `u16` because someone reordered for "natural padding") -- P1 and P2 fail loudly across all 54 samples. Drop the `[[=test_reject_short]]` marker because the parser is now used in a context that pre-checks length, the truncation tests stop wasting CI time. Pin a regression value via `[[=test_examples{...}]]` after a CVE was disclosed; that exact byte appears in every future run regardless of how the random generator drifts. The schema declares the contract; the runner enforces it; nobody re-states the field list in test code.

The pattern slots under any test framework: the per-member loop becomes a Catch2 `SECTION`, a GoogleTest `TEST_P`, or a doctest `SUBCASE`. The reflection-driven runner doesn't replace the framework -- it replaces the per-target boilerplate that's required to write structural property tests by hand.

### Where reflection-alone falls short

Reflection in C++26 is **structural traversal** -- walk the members, read their types, read their annotations, generate code per member. That is exactly the right shape for the round-trip property test: every claim is "for each field, do X" and reflection writes the loop.

It is the wrong shape for **behavior synthesis**. Tools like GoogleMock generate per-method intercepting wrappers that match arguments, sequence expectations, and dispatch actions. You can simulate a thin slice of that today by walking a struct of `std::function` members and replacing each with a recording stub -- but the result is strictly weaker than what GMock already does well, and lacks per-call argument matchers, cardinality matchers, sequenced expectations, and action customization. The reason is structural: today's reflection lets you *read* the interface but not *generate new code* alongside it. The full mock-synthesis story wants C++29 token injection (P3294); it is sketched in "Where this is heading" below, not pretended into existence here.

What reflection-alone today is NOT a substitute for:
- **GoogleMock / Trompeloeil for behavior mocks** -- argument matchers, sequenced calls, action wiring stay better in those frameworks. Reflection earns its keep on schema-driven property tests, not on call-mocking.
- **Behavioural tests** -- "this function returns the median of its inputs" is a logical property, not a structural one. Reflection sees the signature; the property is the developer's contract.
- **Test scheduling / parallelism** -- run order, fixture lifetime, parallel execution stay the framework's job.
- **Coverage measurement** -- reflection generates the test, the framework runs it, but `gcov` / `llvm-cov` measure what actually executed. Three separate concerns.

## Where this is heading

Two things land between now and C++29 that close the gap further.

**C++26 contracts ([P2900](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2024/p2900r6.pdf), Doumler / Garcia / Halpern / Snyder / others).** A contract is a precondition or postcondition attached to a function declaration that the compiler enforces:

```cpp
auto sqrt(double x) -> double
    pre (x >= 0.0)
    post (r: r * r <= x + 1e-9);
```

The contract IS a partial test specification. Once contracts ship, a test runner can read them back via reflection, generate inputs that satisfy the precondition, run the function, and assert the postcondition. The contract is written ONCE on the function; the test runner is generic. A `pre`-violation in production becomes a `pre`-violation in CI long before -- contracts move test specs from the test file into the function declaration.

Status as of 2026-05-04: P2900 is in C++26. Compiler implementations are partial; clang-p2996 honors the syntax but enforcement is opt-in via `-fcontracts=enforce`. GCC 16.1 emits warnings only.

**C++29 token injection ([P3294](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2025/p3294r2.pdf)) is the unlock for behavior synthesis.** Reflection in C++26 lets a generic harness *traverse* an interface; injection lets it *generate code* alongside the interface. That is the gap between today's structural property tests (which work) and today's reflection-driven mocks (which are strictly worse than GoogleMock). With injection, the same `[[=mock_interface{}]]` annotation could trigger generation of a mock class whose methods carry argument matchers, sequenced expectations, and recorded actions -- the things that actually let GMock-style behavior testing work in production. Today that pattern is sketched but not competitive; injection moves it into the same league. Two illustrative shapes:

```cpp
// Pseudo-syntax (P3294 token injection, C++29 target).
// 1. Round-trip harness injected alongside the schema:
[[ =wire_format{version=1, endian=little}, =inject_tests{round_trip} ]]
struct SensorReading {
    std::uint16_t sensor_id;
    std::int32_t  raw_value;
    std::uint8_t  status_flags;
};
// Compiler injects parse() / serialize() / operator== /
// test_round_trip() that exercises every field and the truncated-input
// failure path. Adding a new field auto-extends the test set without
// the developer touching the test file.

// 2. Mock class injected from an interface, on par with what GoogleMock
//    generates from MOCK_METHOD today:
[[ =mock_interface{}, =inject_mock{strict} ]]
struct ILogger {
    virtual void info(std::string_view) = 0;
    virtual void warn(std::string_view) = 0;
    virtual void error(std::string_view) = 0;
};
// Compiler injects MockILogger : ILogger with per-method recorders,
// argument matchers, EXPECT_CALL-equivalent setup and action wiring.
// No MOCK_METHOD lines; the interface IS the mock spec.
```

The state of the codebase one decade out: contracts carry the spec on the function; reflection generates the per-field structural tests today; injection synthesises the harness and the mocks alongside the implementation tomorrow; sanitizers stay in CI to catch the corners none of the above reach. The "we wrote tests but forgot to update them when the schema changed" failure mode -- the most common cause of regression bugs in production -- becomes mechanically impossible.
