> **Library repo:** [github.com/wrocpp/reflect_soa](https://github.com/wrocpp/reflect_soa) · v0.1.0 · MIT.

Prior art: Barry Revzin's 2025 blog post *"Implementing a Struct of Arrays"* showed how reflection turns the SoA pattern from a macro-library chore into a small template. `reflect_soa` productises that and adds two siblings: **AoSoA** (cache-line-aligned blocks) and an **A/B benchmark harness** that picks the layout for you.

## The problem

You have a particle simulation:

```cpp
struct Particle {
    float x, y, z;        // position
    float vx, vy, vz;     // velocity
    float mass;
    std::uint32_t color;
};

std::vector<Particle> particles(100'000);

// Hot loop: update positions
for (auto& p : particles) {
    p.x += p.vx * dt;
    p.y += p.vy * dt;
    p.z += p.vz * dt;
}
```

On a modern x86 this loop is bandwidth-bound. Every particle is 32 bytes; your CPU only cares about 12 of them (position + velocity); the rest evicts cache lines for nothing. The compiler can't auto-vectorise because adjacent position values are 32 bytes apart in memory.

Data-oriented design's answer: SoA.

```cpp
struct ParticleSoA {
    std::vector<float> x, y, z;
    std::vector<float> vx, vy, vz;
    std::vector<float> mass;
    std::vector<std::uint32_t> color;
};
```

Now the hot loop touches only `x/y/z/vx/vy/vz` and nothing else. SIMD auto-vectorises. Bandwidth utilisation 2-3× better.

The catch: you now maintain two types. The `Particle` struct you think about + the `ParticleSoA` bag of parallel arrays. Every change to `Particle` propagates to both.

## The reflection answer

```cpp
#include <reflect_soa/soa.hpp>

struct Particle {
    float x, y, z;
    float vx, vy, vz;
    float mass;
    std::uint32_t color;
};

soa::SoaVector<Particle> particles(100'000);

// Hot loop
for (std::size_t i = 0; i < particles.size(); ++i) {
    particles.x[i]  += particles.vx[i] * dt;
    particles.y[i]  += particles.vy[i] * dt;
    particles.z[i]  += particles.vz[i] * dt;
}

// Or, for auto-vectorisation-friendly range access:
std::span x_span = particles.col<&Particle::x>();
std::span vx_span = particles.col<&Particle::vx>();
for (std::size_t i = 0; i < x_span.size(); ++i)
    x_span[i] += vx_span[i] * dt;
```

`SoaVector<T>` is a `define_aggregate`-synthesised struct whose fields are `std::vector<float> x`, `std::vector<float> y`, and so on — one per reflected field of `Particle`. The storage is the parallel-arrays layout; the access pattern is the positional one.

## AoSoA: cache-line-aligned blocks

For workloads that do *some* whole-particle work (render, cull) and *some* per-component work (integrate), pure SoA hurts the whole-particle operations. AoSoA (Array of Struct of Arrays) strikes a balance: chunks of N particles laid out as SoA within the chunk, but chunks themselves are an AoS.

```cpp
soa::AoSoAVector<Particle, 8> particles_aosoa(100'000);
// Particles stored in groups of 8.
// Each group's x/y/z/vx/vy/vz occupy 8×4 = 32 bytes — one cache line.

particles_aosoa.each_group([dt](auto& group) {
    // group.x is an std::array<float, 8>
    for (std::size_t i = 0; i < 8; ++i) {
        group.x[i] += group.vx[i] * dt;
    }
});
```

Auto-vectorises better than SoA when you're doing small windows at a time, and keeps whole-particle access localised.

## The synthesis

`SoaVector<T>`'s job: walk `T`'s reflection, emit a struct-of-arrays layout. `define_aggregate` is the primitive:

```cpp
template <typename T>
consteval auto build_soa_type() {
    std::vector<std::meta::info> specs;
    constexpr auto ctx = std::meta::access_context::unchecked();
    for (auto m : std::meta::nonstatic_data_members_of(^^T, ctx)) {
        auto field_type = std::meta::type_of(m);
        auto vector_of_field = std::meta::substitute(^^std::vector, {field_type});
        specs.push_back(std::meta::data_member_spec(
            vector_of_field,
            {.name = std::define_static_string(std::meta::identifier_of(m))}));
    }
    return std::meta::define_aggregate(^^soa_impl<T>, specs);
}

template <typename T>
using SoaVector = [: build_soa_type<T>() :];
```

For each `T::field`, emit `std::vector<decltype(T::field)> field` in the synthesised type. That's the whole SoA transform.

## The A/B benchmark harness

You're rarely sure which layout wins for your workload without measuring. `reflect_soa` ships a harness:

```cpp
#include <reflect_soa/bench.hpp>

auto report = soa::bench::compare<Particle>({
    .layouts = {soa::AoS, soa::SoA, soa::AoSoA<4>, soa::AoSoA<8>, soa::AoSoA<16>},
    .n = 1'000'000,
    .workload = [](auto& container, float dt) {
        for (std::size_t i = 0; i < container.size(); ++i) {
            container.x[i] += container.vx[i] * dt;
            container.y[i] += container.vy[i] * dt;
            container.z[i] += container.vz[i] * dt;
        }
    },
});

std::println("{}", report);
//   AoS             14.2 ms/iter  (baseline)
//   SoA              4.3 ms/iter  (0.30× — 3.3× faster)
//   AoSoA<4>         4.1 ms/iter  (0.29× — 3.4× faster)
//   AoSoA<8>         3.8 ms/iter  (0.27× — 3.8× faster, best)
//   AoSoA<16>        4.0 ms/iter
```

The harness generates all four containers from the same `Particle` reflection, runs the workload against each, reports timings + ratios. Your layout decision has evidence instead of folklore.

## When not to use this

- **Hot loops are small / rare.** If you only touch particles occasionally, AoS is simpler and the perf diff is noise.
- **You need per-particle invariants.** AoS keeps `particle.x` and `particle.y` adjacent; debugging a scrambled AoSoA is harder when you don't know the layout.
- **`define_aggregate` is too experimental for your project's compiler pin.** The capability was behind `-freflection-latest` in early clang-p2996; production-grade may need to wait for GCC / MSVC implementations to catch up. Keep an eye on the project's "known-good compiler versions" table.

## What the library ships

Under [github.com/wrocpp/reflect_soa](https://github.com/wrocpp/reflect_soa):

- `soa::SoaVector<T>` — synthesised struct-of-arrays.
- `soa::AoSoAVector<T, N>` — array-of-struct-of-N-arrays.
- `soa::bench::compare<T>({.layouts = ..., .workload = ...})` — harness.
- `soa::span_of<&T::field>(container)` — get an `std::span<field_type>` view.
- Compile-time asserts that every layout produces identical numerical output for the same workload (no silent drift).

v0.1.0 handles flat aggregates. v0.2 roadmap: nested SoA (SoaVector where fields are themselves SoA-transformable), GPU / SYCL tuned layouts (padding for SPIR-V), packed-bitfield variants.
